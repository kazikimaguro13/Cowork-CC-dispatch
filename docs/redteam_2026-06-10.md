# CCD レッドチームレビュー（Fable 5、2026-06-10）

- **対象**: v0.25.0 / `ccd/guard.py`・`ccd/nightly.py`（FixLoop/Integrator/R5）・隔離・オーケストレーション
- **視点**: 「修正係（dispatch 先の Opus）が**インチキ修正を通そうと敵対的に振る舞う**」前提で、guard 5ルールと検証層をすり抜ける経路を探す
- **読み方**: 各項に 深刻度 / 機序 / PoC 概略 / 対策spec種。深刻度は「auto モードで無人 merge まで到達しうるか」で評価
- **総評**: 設計思想（自己申告を信用しない・偽陰性を許さない）は正しく、guard は「指示でなく強制」を実装できている。ただし**強制の網が、思想が想定する範囲より狭い**。特に (A) R2 のテスト無効化検知が decorator 形しか見ない、(B) 自己保護 denylist が integrity-critical モジュールを取りこぼす、の2点は auto モードで実害に至りうる。

---

## 🔴 高 — RT-1: `pytestmark` / collection hook による R2 バイパス

**機序**: `guard._find_skip_markers` は4つの正規表現しか持たず、すべて `@`(decorator) 形を要求する（`@pytest.mark.skip` 等）。テストを無効化する以下の形は**1つも捕まらない**:

- モジュール変数 `pytestmark = pytest.mark.skip(reason=...)` — そのテストファイル全体を skip
- 新規 `tests/conftest.py`（append-only と skip-decorator スキャンは通るが）に `collect_ignore = ["test_protocol.py"]` や `pytest_collection_modifyitems` フックを書いてテストを**収集から除外**

`_is_test_file` は `tests/` 直下の `.py` を append-only 扱いにするが、**収集制御コードを検査するルールが無い**。

**PoC 概略**: 修正係が「標的テストを直す」と称し、新規 `tests/test_auto_xxx.py` の冒頭に `pytestmark = pytest.mark.skipif(True, reason="flaky")` を置き、その下に常に green な見せかけテストを書く。R2 は decorator 形でないので素通り。R4（suite green）は skip されたテストを「失敗ではない」と扱うので green。R5 だけは標的ミュータントが実際に殺せれば pass するが、collection hook 版なら標的テスト自体を収集から外して mutmut の母集団を歪められる余地がある。

**対策spec種（spec_043 候補・guard 強化）**: `_find_skip_markers` に (1) 代入文 `^\s*pytestmark\s*=` (2) `collect_ignore` (3) `pytest_collection_modifyitems` / `pytest_ignore_collect` (4) import エイリアス（`from pytest import mark` → `@mark.skip`、`from unittest import skip` → `@skip`）を追加。**ただし正規表現の追いかけっこは原理的に負ける**ので、本筋は「R4 をグリーン判定でなく**収集テスト数・実行数が修正前から減っていないこと**の検査に格上げ」（skip/deselect は実行数の減少として必ず観測できる — 文字列でなく事実で捕まえる）。これは guard 思想（diff の語でなく事実を見る）と完全に整合。

---

## 🔴 高 — RT-2: 自己保護 denylist が integrity-critical モジュールを取りこぼす

**機序**: `DENYLIST_GLOBS` が守るのは `guard.py` / `nightly.py` / `nightly_*` / `scheduler*` / CI / packaging / profile TOML のみ。しかし v2→v3 で**ループとガードの実装は複数モジュールに分散**した。以下は denylist 外:

- `ccd/loop.py`（FixLoop 本体・無進捗検知）
- `ccd/retry.py`（retryable 境界分類）
- `ccd/translate.py`（テンプレ翻訳 =「AI判断ゼロの剛体」のはずの部分）
- `ccd/metrics.py`・`ccd/backfill.py`（正直メトリクスの計算）
- `ccd/discover.py`・`ccd/adversarial.py`（発見オラクル）
- `ccd/profile.py`（バリデーション = 安全境界の入口）
- Integrator 実体（`nightly.py` 内なので今は守られているが、spec_040 で `ccd/integrate.py` に切り出す選択肢があり、切り出すと**保護が外れる**）

テンプレA（tests-only・R1 allowlist=`tests/`）では production を触れないのでこの穴は閉じている。しかし**テンプレB が有効化されると**、finding の「名指し production ファイル1つ」が上記のいずれかに一致した場合、R1 は許可し denylist は素通りする。つまり「ループが自分の翻訳器やメトリクスや無進捗検知を書き換える」修正が原理的に通る。

**PoC 概略**: テンプレB の adversarial finding が `ccd/metrics.py` のパーサ堅牢化を要求 → 修正係が「ついでに」`convergence_rate` の母集団定義を緩める diff を混ぜる → R1 allow（metrics.py は名指し対象）・denylist 外・R3 は60行以内なら pass → メトリクスが自分に都合よく改変される。「メトリクスが嘘をつく」をシステム自身が再生産する最悪の自己言及バグ。

**対策spec種**: denylist を**ホワイトリスト反転**に再設計する — 「ループが触ってよいのは `tests/` と、テンプレBで名指しされた non-core production ファイルのみ。`ccd/` のうち core（guard/nightly/loop/retry/translate/metrics/backfill/discover/profile/integrate/protocol/dispatch）は**常時 denylist**」。core 集合は1箇所で定義し、新モジュール追加時に core 判定を強制するテストを置く（denylist の自己保護を「列挙」でなく「既定 deny + 明示 allow」に）。

---

## 🟡 中 — RT-3: R5 mutation オラクルの「殺せた理由」を問えない

**機序**: `_verify_r5` テンプレA は `recheck_mutation(...) == "killed"` だけを見る。mutation testing は「原本で pass・ミュータントで fail」を killed とするが、**ミュータントで fail した理由が「正しい assert」か「付随的な例外/タイミング」かを区別できない**。フレーキーなテストや、ミュータント時だけ別経路で例外を踏むテストは、標的挙動を検証していなくても killed を出しうる。

**深刻度が中どまりの理由**: 常時 green な `assert True` 系は原本でも pass・ミュータントでも pass なので survived 判定になり、killed を偽装できない。悪用には「ミュータント時だけ非決定的に落ちるテスト」を仕込む必要があり難度が高い。ただし**無人 auto では「たまたま落ちた」を killed と誤認する偽陽性**の自然発生もありうる。

**対策spec種**: R5 を2回（決定性確認の N 回）実行して killed が安定するかを見る（DESIGN §9.5 ルール5の「決定的・N回」は翻訳器側の文言にはあるが、R5 実装が N 回回しているか要確認 → 回していなければ追加）。+ 追加テストが標的 finding の `signature` に言及/到達しているかの弱いリンク検査（報告専用でよい）。

---

## 🟡 中 — RT-4: R2 の対象が `tests/**.py` に限定（テスト所在の前提）

**機序**: `_is_test_file` は `parts[0] == "tests"` かつ `.py` のみ。doctests・`src/**/tests/`・リポジトリ直下の `test_*.py`・`tests/data/*.json` 等のデータ駆動 fixture は R2 の append-only 保護外。CCD は全テストが `tests/` 直下なので現状は無害だが、**テスト配置を変えた瞬間に保護が静かに外れる**構造依存。

**対策spec種**: 「テスト収集の真実 = pytest の収集結果」を基準にする（RT-1 対策と合流）。静的パス判定でなく「収集されるテストの集合・数」を pre/post で比較すれば配置非依存になる。

---

## 🟡 中 — RT-5: Integrator は live で R5 を再検証しない

**機序**: spec_040 §2-2 は live 再検証を「guard + 既存スイート全緑」と規定。R5（mutation kill）は**クローン内でのみ検証され、apply 後の live では再確認されない**。非3way apply が「クローンと同一に当たるか失敗するか」を保証するため実害は小さいが、「検証した object と merge する object が同一」の保証が apply の決定性に全依存している。

**対策spec種**: 低優先。apply 後に `git diff main..HEAD` を取り、**worker が検証した diff とバイト一致するか**をハッシュ照合する1行ガードを Integrator に追加すれば、「検証済み patch とは違うものが merge される」を構造的に排除できる（偽陰性ゼロ志向の安価な補強）。

---

## 🟢 低 — RT-6: テンプレA に diff サイズ上限が無い

**機序**: R3（サイズ境界）はテンプレB のみ。テンプレA は tests-only・append-only だが**行数無制限**。5000行のノイズテスト追加が pass しうる（suite green は必要なので実害は希薄、レビュー負荷・リポジトリ汚染の DoS 寄り）。

**対策spec種**: テンプレA にも緩い上限（例: +300行）を入れ、超過は HALT でなく朝レポート警告に留める（偽陽性コストを抑える）。

---

## 🟢 低 — RT-7: skip マーカー正規表現の import エイリアス漏れ

**機序**: RT-1 の一部。`from pytest import mark` → `@mark.skip`、`from unittest import skip` → `@skip` が4本の正規表現に当たらない。RT-1 の「実行数ベース検査」へ移行すれば自然に解消するため独立対応は不要。

---

## まとめ — 対策の優先順位

1. **RT-1 + RT-4 + RT-7 を一本化** → 「R4 を green 判定から**収集・実行テスト数の非減少**判定に格上げ」（spec_043: guard 動的検証の強化）。skip/deselect/collection-hook の全形を**事実（実行数減少）**で捕まえる。文字列マッチの追いかけっこを終わらせる。
2. **RT-2** → denylist を「既定 deny + 明示 allow」に反転し core モジュール集合を恒久保護（spec_044: 自己保護の網羅化）。テンプレB 有効化の**前提条件**にする。
3. **RT-3 / RT-5** → R5 の N 回決定性 + Integrator の検証済み-diff ハッシュ照合（spec_045: 検証の堅牢化、低優先）。
4. RT-6 は機会があれば。

**評価の結論**: 致命的な「いま auto で抜かれる」穴は無い（テンプレA 運用かつ全テストが `tests/` 配下、という現運用が偶然 RT-1/RT-2 の発火条件を塞いでいる）。しかし**それは設計の堅牢さでなく運用の狭さによる安全**であり、テンプレB 有効化・テスト配置変更のどちらかで露出する。記事の §5「自分のガードに自分が捕まった日」に対する続編として、「**ガードを敵対的に攻めたら、強制の網が思想より狭かった**」は非常に誠実で強い節になる。RT-1 対策（文字列でなく実行数）はこのプロジェクトの一貫テーマ「自己申告でなく事実を見る」の最も純度の高い実例。
